For any Blitzy work item producing a pull request against an Agent Action Plan, a multi-phase review MUST run as an isolated process after the code generation run declares completion, and MUST NOT overlap with or be performed by the generation run. The review repeats in cycles until the final reviewer issues APPROVED.
The Agent Action Plan MUST include CODE_REVIEW.md at the repository root as a required deliverable created by the code generation run; the review process owns its contents and commit cadence.
Each cycle MUST begin with a pre-flight gate that passes only if all Agent Action Plan deliverables exist, the project builds clean, all tests pass, all static analysis gates pass, and no production-path method returns a placeholder stub. Any failure MUST return the work item to code generation.
CODE_REVIEW.md MUST be committed before the first phase of each cycle, re-committed after every phase state change and after the final verdict, and present in the pull request's final commit. It MUST assign every changed file to exactly one primary domain — Infrastructure/DevOps, Security, Backend Architecture, QA/Test Integrity, Business/Domain, Frontend, or Other SME — and MAY list secondary domains whose specialists review the relevant surface.
Phases MUST execute in fixed order — Infrastructure/DevOps → Security → Backend Architecture → Business/Domain → Frontend → QA/Test Integrity → Other SME — owned by one specialist reviewer each who reviews only and MUST NOT modify code, run fixes, or re-run tests. Every phase MUST run in every cycle and MUST resolve to exactly APPROVED or BLOCKED with findings at file and line specificity.
If any phase resolves BLOCKED, the review MUST NOT terminate that cycle early; every remaining phase MUST still execute and record its verdict. At cycle end, every BLOCKED finding from every phase MUST be aggregated into a single consolidated remediation list in CODE_REVIEW.md, ordered by phase and grouped by file, and the work item MUST be returned to code generation with that list as the complete remediation scope — partial remediation, scope reduction, and reviewer-side fixes are prohibited. Code generation MUST address every finding in the list before declaring completion, and the next cycle MUST restart from the pre-flight gate with all prior approvals, verdicts, and findings discarded.
After every phase resolves APPROVED, a final reviewer MUST confirm pre-flight conditions still hold against the current PR head and that no non-CODE_REVIEW.md commits have been added by the review process since the pre-flight gate, then issue exactly APPROVED or BLOCKED — no qualified verdicts.
Verification: The pull request's final commit contains CODE_REVIEW.md at the repository root with a final reviewer verdict of APPROVED; each cycle records one entry per domain phase in fixed order, each resolved to exactly APPROVED or BLOCKED; every BLOCKED cycle's CODE_REVIEW.md contains a consolidated remediation list covering findings from every BLOCKED phase in that cycle; review activity timestamps fall after the last code generation commit of the cycle being reviewed.
Scope: All Blitzy code generation work items producing a pull request that references an Agent Action Plan with required deliverables.
